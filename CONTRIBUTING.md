# Contributing Guide  
## InvestCalc — учебный групповой проект

Спасибо за желание внести вклад!

## 1. Правила участия
- Уважайте участников.
- Все изменения через Pull Request.
- В ветку `main` коммитит только преподаватель.

## 2. Workflow
Ветки:
- `main`
- `dev`
- `feature/<name>`
- `fix/<name>`
- `docs/<name>`

## 3. Коммиты
Используйте формат:
- `feat: ...`
- `fix: ...`
- `docs: ...`
- `refactor: ...`

## 4. Стандарты кода

### 4.1 Общие
- Читаемые имена переменных и функций.
- Минимум дублирования.
- Логика разделена по слоям (API → Service → Domain → Infrastructure).

### 4.2 Python
- PEP8.
- Структура:
  - `api/`
  - `services/`
  - `models/`
  - `utils/`

### 4.3 JavaScript / TypeScript
- `camelCase` для переменных и функций.
- `const` / `let` вместо `var`.
- Структура:
  - `src/api`
  - `src/services`
  - `src/models`

### 4.4 C#
- PascalCase для классов и методов.
- camelCase для переменных и параметров.
- Имена файлов совпадают с именами классов.
- Одна ответственность на класс (SRP).
- Рекомендуемая структура:
  - `/Api`
  - `/Services`
  - `/Domain`
  - `/Infrastructure`
  - `/Reports`
- Использовать:
  - Dependency Injection
  - DTO-модели для входных/выходных данных
  - async/await
- Избегать:
  - бизнес-логики в контроллерах (только вызовы сервисов)
  - статических сервисов.

## 5. Тестирование
- API тестируется через Postman (см. `project/documents/testing`).
- Формулы проверяются вручную и автотестами.
- Docker-запуск должен работать без ошибок.

## 6. Как сообщать о проблемах
- Создайте Issue на GitHub.
- Опишите шаги воспроизведения и ожидаемый результат.
