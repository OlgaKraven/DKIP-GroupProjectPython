Групповой проект: «Инвестиционный Аналитик ИС»

Мини-информационная система для оценки экономической эффективности других информационных систем

1. Общая идея проекта

Студенты создают мини-информационную систему, которая:

  • принимает входные данные об ИС (затраты, обучение, подписка, период анализа);
  • рассчитывает TCO, ROI, Payback Period по локальному и облачному сценариям (как в ЛР6);
  • проводит анализ чувствительности (±20%);
  • сравнивает сценарии и формирует автоматический отчет;
  • позволяет загрузить несколько ИС и сравнить их инвестиционную привлекательность;
  • имеет REST API или минимальный интерфейс;
  • разворачивается через Docker;
  • хранится в GitHub с грамотной структурой, README и релизом.

2. Легенда проекта

Команды представляют агентство цифрового консалтинга DataForge. К агентству обращаются компании, которым необходимо рассчитывать экономическую эффективность различных информационных систем: ERP, CRM, HelpDesk, сервисов бронирования, систем учета, мобильных приложений и других решений.

У DataForge отсутствует единая платформа для оценки окупаемости, поэтому команды создают сервис InvestCalc, позволяющий:

  • вводить данные по системе;
  • сравнивать локальные и облачные сценарии;
  • рассчитывать чувствительность;
  • формировать готовый аналитический отчет (HTML/PDF);
  • использовать сервис как внутренний продукт агентства.

Итог проекта: рабочий сервис, который можно показать заказчику.

3. Предметные области

Каждая команда выбирает свою тематическую область. Команда получает «клиента» из определенной отрасли, которому требуется рассчитать эффективность внедрения ИС. Предметная область сообщается преподавателю и фиксируется.

    • /calc/sensitivity

  • форма ввода данных;

4.3. Генератор отчёта

  • HTML → PDF (опционально);

  • Dockerfile;

5. План демонстрации по занятиям

5.1. Занятие 1 — Постановка задачи и разработка модели

Цели:

  • выбрать предметную область;
  • определить бизнес-проблему;
  • собрать исходные данные;
  • подготовить модели расчёта (по ЛР6);

Результат:

Демонстрация:

  • файл с вычислениями.

Результат:

  • архитектура;

Демонстрация:

Цели:

Результат:

  • docker-контейнер;
  • отчёт;
  • сценарный анализ.

Демонстрация:

  • запуск контейнера;
  • отчет.

5.4. Занятие 4 — Финальная сборка и презентации

Цели:

  • локальный и облачный сценарий;

  • README;
  • коммиты;

  • пример данных.

7.5. Рабочий прототип — 1 балл

8. Формирование команд

Для выполнения проектной работы студенты объединяются в команды численностью от 2 до 4 человек.

Каждая команда самостоятельно определяет:

  • распределение ролей (аналитик, разработчик, архитектор, дизайнер, специалист по Docker и т.д.);
  • формат взаимодействия;
  • инструменты коммуникации и разработки.

Команда несёт коллективную ответственность за результат, роли могут распределяться гибко.

9. Примечание для команд

Команды могут использовать любые инструменты, технологии, фреймворки, библиотеки и источники, которые считают необходимыми для реализации проекта, за исключением обращения за помощью к преподавателю в ходе выполнения задания.

Допустимые ресурсы включают:

  • документации, мануалы, официальные сайты технологий;
  • форумы, сообщества, GitHub, StackOverflow;
  • генераторы документации, линтеры, тестовые наборы;
  • инструменты моделирования (Miro, Draw.io, Figma, PlantUML);
  • любой софт для разработки (PyCharm, VS Code, IntelliJ, WebStorm и др.);
  • системы для командной работы (Discord, Trello, Notion и др.).

Язык разработки: Python, JavaScript/TypeScript, C#, Java, Go, PHP, Kotlin и любые другие по выбору команды. Главное условие — финальный прототип должен быть рабочим, запускаться с помощью Docker (если выбран Docker) и быть понятным при проверке.

10. Интерактивные примеры (вкладки)

Ниже приведены примеры, которые студенты могут использовать как ориентир при реализации сервиса.

POST /calc/local HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{
  "project_name": "CRM для отдела продаж",
  "capex": 500000,
  "opex": 120000,
  "training_costs": 80000,
  "period_years": 3,
  "expected_savings_per_year": 300000,
  "expected_revenue_growth_per_year": 200000
}

Ожидаемый ответ (пример):

{
  "project_name": "CRM для отдела продаж",
  "tco": 960000,
  "roi": 156.25,
  "payback_period_years": 1.5
}
GET /calc/sensitivity?scenario=cloud&delta=0.2

Пример формата ответа:

{
  "base_case": {
    "tco": 750000,
    "roi": 130.0
  },
  "minus_20": {
    "tco": 720000,
    "roi": 110.0
  },
  "plus_20": {
    "tco": 780000,
    "roi": 150.0
  }
}

Задача команды — продумать, какие именно параметры менять (затраты, выгоды или оба блока) и как визуализировать результаты в отчете.

Пример минимальной связки Dockerfile и docker-compose.yml для Python-сервиса (Flask/FastAPI — по выбору команды):

# Dockerfile (пример)
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "main.py"]
# docker-compose.yml (пример)
version: "3.9"
services:
  investcalc-api:
    build: .
    ports:
      - "8000:8000"
    restart: unless-stopped

Команда может адаптировать пример под выбранный стек (Node.js, Java, C#, Go и др.), главное — добиться запуска сервиса одной командой docker compose up.
